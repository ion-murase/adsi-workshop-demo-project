# コーディングスタイル（共通）

> 言語固有の規約は [java-spring-boot.md](../java-spring-boot.md) と [typescript-frontend.md](../typescript-frontend.md) が優先する。

## イミュータビリティ（重要）

既存オブジェクトを変更せず、新しいオブジェクトを作る:

- 副作用のないデータフローを優先する
- デバッグしやすく、並行処理も安全になる

## 基本原則

### KISS（Keep It Simple）

- 動く最も単純な解決策を選ぶ
- 早すぎる最適化を避ける
- 賢さより読みやすさを優先する

### DRY（Don't Repeat Yourself）

- 重複ロジックは共通化する
- コピペによる実装の乖離を防ぐ
- 抽象化は「本当に繰り返されている」ときだけ行う

### YAGNI（You Aren't Gonna Need It）

- 必要になる前に機能や抽象を作らない
- 先回りした汎用化を避ける
- シンプルに始め、圧力が来たらリファクタする

## ファイル構成

- 小さく凝集したファイルを優先する（目安 200–400 行、上限 800 行）
- 機能・ドメイン単位で整理する（ファイル種別だけでの分類は避ける）
- 大きくなったらユーティリティやサブモジュールへ切り出す

## エラーハンドリング

- すべての境界でエラーを明示的に扱う
- UI 向けにはユーザーに分かるメッセージを返す
- サーバー側には詳細なコンテキストをログに残す
- エラーを握りつぶさない

## 入力バリデーション

- ユーザー入力はシステム境界で必ず検証する
- スキーマベースのバリデーションを使う
- 外部データ（API レスポンス、ファイル内容）は信頼しない

## 命名規則（共通）

- 変数・関数: `camelCase`（意味が伝わる名前）
- 真偽値: `is`, `has`, `should`, `can` プレフィックス
- 型・インターフェース・コンポーネント: `PascalCase`
- 定数: `UPPER_SNAKE_CASE`

## 避けるべきコードの臭い

- **深いネスト**: 早期 return で浅く保つ
- **マジックナンバー**: 名前付き定数を使う
- **長い関数**: 責務ごとに分割する（目安 50 行以内）

## 完了前チェックリスト

- [ ] 読みやすく、名前が意図を表している
- [ ] 関数が小さい（<50 行）
- [ ] ファイルが焦点を絞っている（<800 行）
- [ ] ネストが浅い（>4 レベルになっていない）
- [ ] エラーハンドリングがある
- [ ] ハードコード値がない（定数・設定を使う）
- [ ] ミューテーションを避けている
